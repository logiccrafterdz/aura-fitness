# AURA Fitness Club — Replit Execution Plan

## Objective
Use the 24-hour free Replit Agent window to build the AURA system as aggressively as possible while keeping the codebase structured, maintainable, and ready for continued development.

## Delivery Strategy
The goal is not a tiny MVP. The goal is a broad v1 foundation with:
- end-to-end implementation for core operational modules,
- integration-ready scaffolding for advanced external dependencies,
- a clean architecture that can continue after the Replit sprint.

## Product Scope to Target
### Fully implement
- Authentication
- Roles and permissions
- Admin shell and navigation
- Members / CRM
- Plans and memberships
- Invoices and payments
- Cash and manual transfer confirmation flow
- QR access workflow
- Access logs
- Classes and bookings
- Notification templates and event triggers
- Reporting dashboards
- Settings and audit logs
- Staff basics
- Store / POS basics

### Integration-ready only
- NFC native behavior
- MQTT hardware sync with real controllers
- Full CIB/Edahabia live gateway integration
- Predictive analytics
- Wearables, chatbot, live streaming
- Full native mobile app if time becomes constrained

## Best Technical Shape
### Preferred repo structure
- `apps/admin` — Next.js admin panel
- `apps/api` — NestJS API
- `packages/db` — schema, migrations, seed
- `packages/shared` — shared types, schemas, constants
- `docs/` — product and execution docs

### Architecture style
- Modular monolith
- One database
- Domain modules with clean boundaries
- Shared validation and DTO discipline
- Seeded demo environment

## Practical Acceptance Criteria
### Foundation accepted when
- App boots
- API runs
- DB migrates
- Admin auth works
- Roles are enforced

### Members accepted when
- Create, update, list, detail, filter members works
- Membership status is visible and editable
- Plans can be assigned

### Billing accepted when
- Invoices can be created and listed
- Payments can be registered
- Cash and manual transfer flows both exist

### Access accepted when
- QR can be generated
- Verification endpoint returns allow / deny + reason
- Access attempts are logged

### Classes accepted when
- Sessions can be created
- Members can be booked in
- Capacity and waitlist basic rules work

### Reporting accepted when
- Dashboard shows real counts and sums
- At least a few operational reports exist

## Replit Tactics
- Make Agent plan first.
- Force implementation in large but bounded chunks.
- Ask for summaries after each chunk.
- Periodically ask for a whole-project bug sweep.
- Do not let Agent drift into unrelated redesigns.

## Success Definition
By the end of the build window, the project should look like a serious operating system for the gym, not a speculative prototype.
