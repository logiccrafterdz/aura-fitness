# AURA Fitness Club — Replit Build Instructions

This file is the execution contract for Replit Agent.
The source of truth for product scope is `docs/source-ts.md`.

## Mission
Build as much of the full AURA Fitness Club system as possible in a production-structured way during the intensive build session.
Do not build a toy demo. Build a real foundation that can continue after the session.

## Source of Truth
Read `docs/source-ts.md` fully before planning or coding.
If this file conflicts with your assumptions, prefer the source document and this file's execution constraints.

## Product Goal
AURA is a local-first, offline-aware fitness club management platform tailored to the Algerian market.
Core domains include:
- Membership & CRM
- Plans, memberships, invoices, and payments
- QR-based access control with logs and policy rules
- Classes and bookings
- Staff and roles
- Store/POS
- Notifications
- Reporting and dashboards
- Settings, audit, and security foundations

## Build Goal for This Session
Create a complete, organized codebase that covers the platform broadly.
Prioritize real implementation for core modules and integration-ready implementation for advanced external dependencies.

## Mandatory Principles
- Read first, plan first, then code.
- Use a production-oriented structure.
- Keep modules clean and typed.
- Favor shipping broad functional coverage over polishing edge features too early.
- Never refactor unrelated working code without reason.
- After each major step, summarize what changed, what remains, and any blockers.

## Recommended Stack
- Admin web: Next.js + TypeScript
- API: NestJS + TypeScript
- Database: PostgreSQL
- Cache / queue-ready layer: Redis only if needed
- Validation: Zod or class-validator consistently
- ORM: Prisma or Drizzle, but use one consistently
- UI: clean admin UI, responsive, no visual over-engineering

## Architecture Rules
- Prefer a modular monolith.
- Separate domains clearly: members, billing, access, classes, store, staff, notifications, reporting, settings.
- Shared types and schemas should be centralized.
- Add seed data and realistic demo records.
- Add environment variable examples.
- Add setup instructions.

## Priority Order
1. Project structure, auth, RBAC, database, audit log, settings
2. Members / CRM
3. Plans, memberships, invoices, payments
4. Access control with QR and access logs
5. Classes and bookings
6. Notifications templates and event triggers
7. Reporting dashboard
8. Store / POS basic
9. Staff / shifts basic
10. Loyalty basic if time allows

## What Must Be Fully Implemented
- Authentication and role-based access
- Admin dashboard
- Member CRUD and membership lifecycle
- Plans and pricing management
- Invoice and payment records
- Cash payment flow
- Manual Baridimob proof-of-payment flow
- QR generation and access verification endpoint
- Access logs and basic time-rule checks
- Class/session/booking basics
- Notification templates/events basics
- Reporting pages with real queries
- Settings and audit log pages

## What Can Be Integration-Ready
- Live payment gateways
- NFC hardware integrations
- MQTT hardware sync with real devices
- Advanced analytics / forecasting
- External BI integrations
- Native mobile-only capabilities
- Wearables / chatbot / live streaming

## Non-Goals During Initial Build Burst
- Pixel-perfect final branding
- Native Flutter app in the same first pass unless the core web platform is already stable
- Complex microservices split
- Overbuilt DevOps before the product works

## Working Style
Always use this loop:
1. Read relevant docs
2. Produce or update plan
3. Implement one major block
4. Run / validate / fix
5. Summarize progress
6. Move to next block

## Quality Bar
- No fake data flows without explicit labels
- No dead pages with no backing logic unless clearly marked planned
- No broken imports or obvious type errors
- No duplicated schemas across layers unless necessary
- Keep naming consistent and domain-driven

## Deliverables Expected by End of Session
- Working repo with admin app and API
- Database schema and seed
- Main admin flows operational
- Demo accounts and sample records
- Documentation for running and continuing development
